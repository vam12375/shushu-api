# 公益站防滥用功能 — 实施总结

> 适用场景：在 L 站（linux.do）公开分享的公益 API 站，防止 APIKEY 被共享、倒卖、多人滥用。
> 实施日期：2026-06-05

---

## 一、需求与实现对照

| 需求 | 实现 | 触发后果 |
| --- | --- | --- |
| 每个用户最多创建 1 个 APIKEY | `controller/token.go:AddToken` 普通用户 `CountUserTokens >= 1` 拒绝（管理员豁免） | 创建被拒，提示先删除现有令牌 |
| 30 分钟内出现 3 个不同 IP → 禁用 APIKEY | `service/ip_guard.go` 滑动窗口；`middleware/auth.go:TokenAuth` 接入 | 禁用该用户全部令牌 + 提示「删除 API 重新创建」 |
| 一天内触发 3 次 → 封禁用户 | strike 计数（24h 窗口）达阈值 | 用户 `Status=disabled` + LinuxDO ID 写入黑名单 |
| 删号重注册也封死 | 以不可变的 **LinuxDO ID** 为锚点写入黑名单表 | OAuth 注册/登录/绑定阶段均拦截 |

## 二、改动文件清单

### 新增
- **`model/ban.go`**
  - `BannedUser` 表：`{ LinuxDOId, UserId, Reason, CreatedAt }`
  - `IsLinuxDOIdBanned(id)` / `AddLinuxDOBan(id, userId, reason)`
  - `DisableUserTokens(userId)`：禁用用户全部令牌 + 清令牌缓存
  - `BanUserByIPGuard(userId, reason)`：禁用用户 + 写黑名单 + 清缓存
- **`service/ip_guard.go`**
  - `CheckIPGuard(userId, clientIP) IPGuardResult`：核心入口
  - Redis 优先（ZSET 滑动窗口 + INCR strike 计数），无 Redis 时进程内存兜底
  - 对主请求完全容错：任何内部错误降级为「放行」，不影响正常转发

### 修改
- **`model/main.go`**：`BannedUser` 注册进 `migrateDB` 与 `migrateDBFast` 两处 AutoMigrate
- **`middleware/auth.go`**：`TokenAuth` 用户状态校验通过后调用 `service.CheckIPGuard`，按结果 abort 并返回中文提示
- **`controller/token.go`**：`AddToken` 增加每用户 1 令牌限制（管理员豁免）
- **`controller/linuxdo.go`**：`LinuxdoOAuth` 与 `LinuxDoBind` 拿到 LinuxDO 用户后查黑名单，命中即拒绝

## 三、可配置项（环境变量）

| 变量 | 默认值 | 含义 |
| --- | --- | --- |
| `IP_GUARD_ENABLED` | `true` | 是否启用 IP 守卫 |
| `IP_GUARD_WINDOW_MINUTES` | `30` | IP 滑动窗口（分钟） |
| `IP_GUARD_DISTINCT_IP_THRESHOLD` | `3` | 窗口内触发禁用的不同 IP 数 |
| `IP_GUARD_STRIKE_WINDOW_HOURS` | `24` | strike 滑动窗口（小时） |
| `IP_GUARD_STRIKE_THRESHOLD` | `3` | 触发封禁的 strike 次数 |

> 默认值即为本次需求数值，无需额外配置即可生效。

`docker-compose.yml` 示例片段：

```yaml
environment:
  - IP_GUARD_ENABLED=true
  - IP_GUARD_WINDOW_MINUTES=30
  - IP_GUARD_DISTINCT_IP_THRESHOLD=3
  - IP_GUARD_STRIKE_WINDOW_HOURS=24
  - IP_GUARD_STRIKE_THRESHOLD=3
```

## 四、工作流程图

```
API 请求 (带 sk-xxx)
   │
   ▼
TokenAuth 校验 token / 用户状态
   │  通过
   ▼
CheckIPGuard(userId, clientIP)
   │
   ├─ 记录 IP 到滑动窗口(ZSET / 内存)
   │
   ├─ 窗口内不同 IP < 3 ──► 放行，正常转发
   │
   └─ 不同 IP ≥ 3
        ├─ 禁用该用户全部令牌
        ├─ strike +1（24h 窗口）
        ├─ 重置 IP 窗口（给「删 key 重建」机会）
        │
        ├─ strike < 3 ──► 返回「请删除 API 重新创建」
        └─ strike ≥ 3 ──► 封禁用户 + LinuxDO 黑名单
                          返回「账号已被封禁」
```

下次该 LinuxDO 账号即使删号重注册 → `LinuxdoOAuth` 查黑名单命中 → 拒绝。

## 五、设计要点 / 项目规则遵循

1. **不可变锚点**：用户记录可软删除，但 LinuxDO ID 永久不变，是封禁的可靠依据。
2. **三库兼容**：`BannedUser` 全部走 GORM 抽象，无原生 SQL，SQLite/MySQL/PostgreSQL 通用（Rule 2）。
3. **容错降级**：IP 守卫的 Redis 失败不阻断主请求，公益站可用性优先。
4. **缓存即时失效**：禁用/封禁后同步清理用户与令牌 Redis 缓存，避免缓存导致封禁延迟生效。
5. **管理员豁免**：1-key 限制仅作用于普通用户，便于站长自测与运维。

## 六、注意事项

- **默认令牌**：若开启 `GenerateDefaultToken`，密码注册时会自动建 1 个令牌，用户即已用满 1-key 配额，符合预期。
- **解封操作**：如需手动解封，删除 `banned_users` 表中对应 `linux_do_id` 记录，并将用户 `status` 改回 `1`。
- **IP 来源**：守卫使用 `c.ClientIP()`，请确保反向代理（Nginx/Cloudflare）正确透传真实 IP（`X-Forwarded-For` / `X-Real-IP`），否则所有用户会被识别为同一代理 IP。
