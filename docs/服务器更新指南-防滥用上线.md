# 服务器更新指南 — 公益站防滥用功能上线

> 架构：用户 → Cloudflare → 宝塔 Nginx → Docker(new-api:3000)
> 部署方式：GitHub Actions 构建镜像推 GHCR，服务器拉取运行。

---

## 一、触发镜像构建（在本地电脑做）

`my-deploy.yml` 不会因普通 push 自动触发，需手动触发其一：

**方式 A — 打 tag（推荐）**
```bash
git tag deploy-anti-abuse-v1
git push origin deploy-anti-abuse-v1
```

**方式 B — 网页手动**
打开 `https://github.com/vam12375/shushu-api/actions/workflows/my-deploy.yml`
→ Run workflow → 选 main → 运行。

等 Actions 出现绿勾（约 5–10 分钟），镜像 `ghcr.io/vam12375/shushu-api:latest` 即更新。

---

## 二、关键前置：宝塔 Nginx 真实 IP 透传（必做，否则 IP 守卫失效）

双层代理（CF + Nginx）下，若不处理，容器只会看到 Cloudflare 的出口 IP，
导致所有用户被识别成同一个 IP —— IP 守卫要么不触发，要么跨用户误封。

**在宝塔 → 网站 → api.faction168.online → 反向代理 → 配置文件**，
在 `location` 块里把 XFF 头改为取 Cloudflare 透传的真实 IP：

```nginx
# 真实用户 IP 来自 Cloudflare 的 CF-Connecting-IP 头
proxy_set_header X-Real-IP        $http_cf_connecting_ip;
proxy_set_header X-Forwarded-For  $http_cf_connecting_ip;
proxy_set_header Host             $host;
proxy_set_header X-Forwarded-Proto $scheme;
```

> 要点：用 `$http_cf_connecting_ip`（Cloudflare 一定会带这个头），
> **不要**用默认的 `$proxy_add_x_forwarded_for`（那样会带上 CF 的 IP，反而被容器当成客户端）。

保存后 `nginx -t && nginx -s reload`（或在宝塔点重载配置）。

---

## 三、更新容器（在服务器做）

进入放 `docker-compose.deploy.yml` 的目录：

```bash
# 1. 确认 compose 里已有 TRUSTED_PROXIES（见下方“四”），如没有先补上
# 2. 拉最新镜像
docker compose -f docker-compose.deploy.yml pull new-api

# 3. 重建容器（启动时自动迁移，建 banned_users 表，无需手动建表）
docker compose -f docker-compose.deploy.yml up -d

# 4. 看日志确认无报错、迁移完成
docker compose -f docker-compose.deploy.yml logs -f --tail=80 new-api
```

> 若你服务器上文件名是 `docker-compose.yml`，去掉 `-f docker-compose.deploy.yml`。

---

## 四、容器环境变量（确认已加）

`new-api → environment` 下应有：

```yaml
      - TRUSTED_PROXIES=172.16.0.0/12,127.0.0.1
```

含义：经过宝塔 Nginx 改写后，容器收到的就是真实用户 IP，
这里只需信任 Docker 内网网关（Nginx 回源到容器走 Docker 内网）。

可选阈值（不填即用默认：30min/3IP 禁用，24h/3次 封禁）：
```yaml
      - IP_GUARD_ENABLED=true
      - IP_GUARD_WINDOW_MINUTES=30
      - IP_GUARD_DISTINCT_IP_THRESHOLD=3
      - IP_GUARD_STRIKE_WINDOW_HOURS=24
      - IP_GUARD_STRIKE_THRESHOLD=3
```

## 五、上线后自检

1. **看启动日志**应出现：
   - `database migration started` → 无 error
   - `已配置可信代理: [172.16.0.0/12 127.0.0.1]`
2. **验证真实 IP 透传**：发一个 API 请求，到「日志」页看记录的 IP 是不是你的真实公网 IP（不是 CF 或内网 IP）。
   - 若显示成 `172.x` / CF 的 IP → 第二步 Nginx 头没配对，IP 守卫会失效，务必修正。
3. **验证 1-key 限制**：普通用户控制台新建第二个令牌应被拒。

---

## 六、误封了怎么办（手动解封）

防滥用用 MySQL，可进 MySQL 容器操作：

```bash
docker compose -f docker-compose.deploy.yml exec mysql mysql -uroot -p你的密码 new-api
```
```sql
-- 查黑名单
SELECT * FROM banned_users;
-- 解封某 LinuxDO 账号（删黑名单 + 恢复用户状态）
DELETE FROM banned_users WHERE linux_do_id = '该用户的linuxdo_id';
UPDATE users SET status = 1 WHERE linux_do_id = '该用户的linuxdo_id';
-- 恢复其被禁用的令牌
UPDATE tokens SET status = 1 WHERE user_id = (SELECT id FROM users WHERE linux_do_id='...');
```
> 用了 Redis 时，解封后建议重启 new-api 容器清缓存，或等缓存自然过期。

---

## 七、首次上线建议

L 站刚分享、流量未知时，可先**只开 1-key 限制、观察 IP 守卫**：
- 把 `IP_GUARD_DISTINCT_IP_THRESHOLD` 暂设大一点（如 5），跑几天看日志里有多少触发，
- 确认没有误伤正常用户（手机/宽带切换会换 IP）后，再调回 3。
- 或先 `IP_GUARD_ENABLED=false`，等真实 IP 透传验证无误后再开。
